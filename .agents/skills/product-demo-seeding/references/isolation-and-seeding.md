# Isolation And Seeding

## Prove Current Source Before Runtime

Fetch `origin/main` and derive capture source from that remote ref, not from the current branch:

```bash
git fetch origin main
git worktree add --detach <capture-worktree> origin/main
git -C <capture-worktree> rev-parse HEAD
git rev-parse origin/main
git -C <capture-worktree> status --porcelain
```

Keep the two SHAs in provenance. Continue only when HEAD matches `origin/main`, the detached worktree is clean, and dependencies/build outputs come from that worktree. Record the fetch time and build commands. A source checkout alone is insufficient if the browser still serves a stale bundle.

Create a sibling artifact root, not a directory under landing/public production assets. Put capture source copies, raw masters, metadata, configs, proofs, delivery candidates, logs, and checksums there. Promotion is a later explicit operation.

Keep temporary capture harness code under that external `CAPTURE_ROOT`. When the E2E runner requires a spec inside the capture worktree, put only that literal path in a take-owned excludes file and pass it to the entire runner with command-local `core.excludesFile` environment configuration. This makes nested Git status checks inherit the same whitelist without changing repository configuration. Verify the clean-worktree status gate remains empty before every take, copy the spec into the staged source bundle, and remove the spec plus exclusion at teardown. Never edit the shared `.git/info/exclude`; linked worktrees share it. A broad ignore pattern or a dirty status hidden only in provenance is not acceptable.

## Pick The Operating Mode

| Need | Mode | Why |
| --- | --- | --- |
| Explore current UI and choose a story | `scripts/dev-isolated --web` | Fast manual iteration with disposable state |
| Produce a repeatable capture | `apps/web/e2e` worker fixture | Owns backend, frontend, database, ports, reset, and teardown |
| Exercise Docker or SSH behavior itself | `containers` E2E project | Uses real container-backed executors |

Do not manually point a browser at a running personal instance. A different browser profile does not isolate the backend or database. Never inspect or stop a developer's instance as part of capture teardown.

## Manual Exploration

From Kandev repo root:

```bash
scripts/dev-isolated --web
```

Use the printed web URL, backend URL, temp home, database, logs, PID file, and teardown command. Confirm none match known production ports or the user's normal Kandev home. Manual exploration is not a recordable take because it lacks per-take state ownership.

Do not pass `--copy-db` for product media. Fresh data is both safer and easier to make coherent.

Before leaving exploration:

```bash
scripts/kandev-kill --pidfile <printed-pidfile> --yes
```

Confirm the printed backend/web ports have no listeners.

## Reproducible Capture Fixture

Build capture specs on `apps/web/e2e/fixtures/test-base.ts`. It provides:

- worker-scoped backend/frontend and unique ports;
- fresh temp `HOME`, `KANDEV_HOME_DIR`, SQLite DB, repositories, and worktree roots;
- `apiClient` for supported setup and test-harness routes;
- `seedData` for a valid workspace, workflow, repository, agent profile, and executor profile;
- reset and teardown.

Choose a free `E2E_PORT_OFFSET`; never paste an offset from old notes without checking listeners. Keep each concurrent desktop/mobile run on a different offset.

Allocate and record all take-owned resources together:

```text
take slug:
source HEAD/origin-main SHA:
HOME / KANDEV_HOME_DIR:
database / repository / worktree root:
backend / agentctl / frontend ports:
X display:
browser profile directory:
artifact root:
PID files and logs:
```

Use an unclaimed X display and a new browser profile directory per take. Probe ports and display sockets before launch, then prove the launched process arguments contain only the recorded temp paths and ports.

Use `data-testid`, role locators, and page objects. Snapshot refs and CSS generated by Radix/Dockview drift too easily for reproducible capture.

## Useful Seed APIs

Read current signatures in `apps/web/e2e/helpers/api-client.ts` before use. Common building blocks:

- `createTask`: create without launching the fixture agent.
- `createTaskWithAgent`: use only when a real mock-agent turn is part of the visible story.
- `updateTaskState`: place tasks in believable workflow states.
- `createRepository`: attach a disposable local Git repository.
- `createExecutor` / `createExecutorProfile`: add valid profiles for selectors.
- `seedTaskSession`: create controlled `RUNNING`, `IDLE`, or completed sessions.
- `seedSessionMessage`: add distinct thinking, tool, result, and response rows.
- mock-provider helpers: seed PRs, checks, reviews, comments, and integration state.
- `e2eReset` and delete helpers: clean scenario-specific records.

Use service APIs for normal product state. Use `/_test` seed routes only for state that would otherwise require slow external processes or nondeterministic timing.

## Deterministic Provider State

Run mock GitHub and Jira providers at the backend boundary. Keep provider credentials unset and prevent live provider network calls; stop when logs cannot prove mock routing. Do not rely on a real account merely because its data is read-only.

Version one fixture document containing fixed IDs and timestamps plus fixed actors, repositories/projects, titles, descriptions, PR/issue states, checks, comments, labels, and sort order. Use credible but fictional copy. A repeated seed must produce the same canonical record IDs and content.

Use current provider APIs and actual integration UI. Do not create a fake detail page when the current GitHub UI only supports inspecting a list row. Jira may use its real detail panel when current main exposes one.

## Rehearsal And Per-Take Reset

Rehearsal must use the same current build, theme preference, route, fixture version, selectors, and dimensions as RECORD, but its database is disposable. After rehearsal, stop its processes and discard or completely reset its state.

Prefer a fresh database before every recorded take. If fixture architecture requires reset-in-place, prove the reset removed every workspace-scoped task, session, import, and provider-created record before applying one canonical seed. Reload the browser after reset; a clean database with a stale client store is not clean capture state.

Pre-record assertions should include:

1. exact counts for workspace, workflow, stage, starting tasks, sessions, and provider objects;
2. unique canonical task IDs and titles;
3. absence of any task title created by rehearsal or a prior take;
4. exact provider row IDs and deterministic ordering;
5. one expected starting task card, not duplicate fixture accumulation;
6. visible UI scan after a hard reload.

Fail the take instead of deleting duplicate cards in the DOM. Retry from a new database/temp root when a recording aborts after product mutation.

## Disposable Repository

Create a fresh local Git repository under the fixture temp root. Give it:

- a fictional project name and reserved/test-only identity;
- a clean `main` commit;
- a feature branch or worktree when the story needs changes;
- small readable source files with meaningful names;
- a nontrivial diff: one behavior change plus one focused test;
- no copied proprietary code, tokens, remotes, or host paths.

Git state should support the UI being shown. A review story needs actual changed files and diff content; an editor story needs actual readable files; a terminal story needs commands that produce truthful output.

## Narrative Coherence

Use one fictional domain per capture batch. Define before seeding:

```text
organization:
project/repository:
workflow stages:
task titles:
feature branch:
changed files:
PR/check/review state:
agent and model labels:
expected result:
```

Avoid generic names such as `E2E Workspace`, `Test Task`, `foo.ts`, and `Mock response`. Avoid claims that imply production integrations or agents were contacted.

## Executor And Integration Honesty

Only show entries returned by current product APIs. To demonstrate Local, Worktree, Docker, Sprites, or SSH, create valid disposable profiles using their actual executor types and safe test-only config. The selector itself must remain product UI.

For integrations, mock provider responses at the backend boundary. Do not inject logos or entries into the page. If current settings expose fewer integrations than marketing copy expects, capture what exists and report the gap.

## Seed Verification

Before recording:

1. Reload the page so boot payload and client store agree.
2. Assert story-critical backend invariants with `apiClient`.
3. Assert every story-critical control and label through the UI.
4. Scan visible text for `E2E`, `mock`, `fixture`, `/tmp`, `/home`, localhost URLs, slash directives, and host usernames.
5. Exercise the complete route once without a recorder against rehearsal state.
6. Discard/reset rehearsal state and prove the canonical record counts again.
7. Record exact provenance and teardown ownership: fixture, PID file, temp root, database, ports, display, browser profile, artifact root, and temporary profiles/specs.

If any invariant depends on timing luck, redesign the seed. Marketing capture should be deterministic.

## Provenance And Teardown

Write a machine-readable manifest before RECORD. Include source/fetched SHAs, clean-status proof, fetch/build commands, fixture/seed hash, fixed clock/timezone, feature/profile environment, process arguments, resource map, route selectors, sanitation, and take slug. Append start/end times and output hashes after capture.

Teardown only recorded take-owned resources. Stop processes through their PID files; verify each recorded port has no listener and each display/browser process is gone; remove temporary capture specs, browser profiles, databases, repositories, and worktrees; retain only the staged evidence and approved artifacts. Record each check and any intentionally retained path. A cleanup command without post-cleanup verification is not proof.

Reject these shortcuts:

- a stale checkout, stale UI bundle, or inherited browser session;
- `--copy-db`, any developer DB/data, or an existing Kandev instance;
- shared ports, display, browser profile, database, or temp root across concurrent takes;
- live provider credentials/network, nondeterministic dates, or provider ordering;
- recording against rehearsal-mutated state or reusing a failed take's DB;
- DOM/CSS patches that hide duplicate data or broken responsive behavior;
- staging raw/proof/delivery files inside production assets.
